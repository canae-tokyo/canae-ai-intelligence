# 現行ステータス

## Design Principles

1. ブランド・URL・GitHub・インフラは独立した4レイヤーとして管理する。
2. AI IntelligenceはWeb Assistの研究・検証部門として位置付けるが、システムは完全に独立させる。
3. 公開ベンチマークとCANAE独自評価を混同しない。
4. AIニュースサイトではなく、AI業界を構造化・可視化する知識基盤を目指す。
5. ニュースよりもCANAEの実務検証を価値の中心に据える。

- `npm run candidate:register`による重複・差分検知済み候補の登録
- `npm run validate:registration`による候補登録レポート検証
- 通常実行はdry-run、`--apply`指定時のみ`data/update-candidates.json`へ原子的に反映する書き込み境界
- Candidate ID、完全`canonicalUrl`、正規化`canonicalUrl`の重複拒否
- `pending` / `draft`固定で、`verified`昇格と正本データ更新を行わない候補登録境界
- `data/update-candidates.json`へGitHub Changelog由来のpending候補1件を登録
- v1.1.2 Manual Review Foundation
- `npm run candidate:review`によるpending候補の手動レビュー結果記録
- `npm run validate:manual-review`による手動レビューレポート検証
- `reviewDecision`追加により既存`reviewStatus` enumとの互換性を維持
- `approved` / `rejected` / `on-hold`の手動判断、保留解除、終端状態、同時更新検知を実装
- 正本データ更新、`verified`昇格、レビューUI、通知を行わない手動レビュー境界
- v1.1.2 Verified Promotion Foundation
- `npm run candidate:promote`による承認済みnews候補の昇格基盤
- `npm run validate:promotion`による昇格レポート検証
- `reviewDecision: approved`かつ`reviewStatus: accepted`のnews候補のみ昇格対象
- `news.json`と`update-candidates.json`の2ファイル原子的更新、入力ハッシュ再確認、バックアップ、ロールバック境界
- news ID、完全URL、正規化URLの重複拒否と必須メタ情報検証
- 昇格後newsの`status` / `dataQuality`を`verified`に固定し、候補側へ昇格先を記録
- v1.1.2 Review UI Foundation
- `/internal/review-candidates`による更新候補の読み取り専用確認画面
- `pending` / `reviewing` / `accepted` / `rejected`のステータス絞り込み
- 候補詳細、外部リンク、`duplicateCheck`、`diffSummary`、レビュー履歴の表示
- 公開ナビ導線、書き込み処理、API route、UI操作による承認・却下・昇格を追加しない安全境界
- Cloudflare Workers Build Version `106a4297-e231-476e-b4be-8ac5a56478bc`
- v1.1.2 Internal Review Access Control Foundation
- Cloudflare Worker境界で`/internal/*`をStatic Assets配信前に保護
- Cloudflare Access JWT、許可メール、Fail-closed、ローカル限定バイパスを実装
- `assets.run_worker_first`により`/internal`と`/internal/*`をWorker先行に固定
- 未認可`/internal/review-candidates`は404を返し、候補情報をレスポンスに含めない
- `npm run validate:access-control`によるアクセス制御検証

## Phase 1 Freeze

Phase 1 / v1.0.1相当は凍結版です。

- 不具合修正、セキュリティ対応、誤記修正、運用維持に必要な最小修正のみ許可する。
- 新機能追加、大幅なUI変更、データ構造変更はv1.1以降で扱う。
- v1.0.1時点の正本文書は保持し、必要な変更は`CHANGELOG`と新バージョン文書で管理する。
- `noindex`解除はv1.0.1凍結版では行わない。
