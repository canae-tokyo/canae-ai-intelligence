# v1.1.2 自動収集・更新補助設計

## Design Principles

1. ブランド・URL・GitHub・インフラは独立した4レイヤーとして管理する。
2. AI IntelligenceはWeb Assistの研究・検証部門として位置付けるが、システムは完全に独立させる。
3. 公開ベンチマークとCANAE独自評価を混同しない。
4. AIニュースサイトではなく、AI業界を構造化・可視化する知識基盤を目指す。
5. ニュースよりもCANAEの実務検証を価値の中心に据える。

## 目的

v1.1.2では、公式一次情報を継続的に収集し、ニュース・ツール・ベンチマーク更新候補を作成するための設計を固定します。

自動化の目的は、公開作業の完全自動化ではなく、候補収集、整形、重複検知、差分検知、レビュー補助です。

## 最重要ルール

自動収集は`verified`へ直接昇格しません。

| 処理 | 自動化可否 |
| --- | --- |
| 公式ソース確認 | 可 |
| 候補作成 | 可 |
| 重複検知 | 可 |
| 差分検知 | 可 |
| `draft`登録 | 可 |
| 重要度判断 | 人手必須 |
| CANAE影響メモ確定 | 人手必須 |
| `verified`昇格 | 人手必須 |
| Production反映 | 人手必須 |

## 対象

| 対象 | 内容 |
| --- | --- |
| `collection-sources.json` | 収集対象ソース一覧 |
| `update-candidates.json` | 収集・差分検知で生成される候補 |
| `npm run validate:collection` | ソース・候補データの検証 |
| 運用フロー | draft登録からverified承認までの手順 |
| 失敗時ルール | 取得失敗、重複、URL不正、差分過多の扱い |

## 非対象

- 外部DB
- ログイン機能
- 外部API連携の本番実装
- LLM自動要約の本番投入
- 自動公開
- 自動`verified`昇格
- `noindex`解除
- 大規模UI変更
- v1.1.x互換性を壊すスキーマ変更

## update-candidates.json

収集・差分検知で作成される候補を管理します。

| フィールド | 説明 |
| --- | --- |
| `id` | 候補ID |
| `sourceId` | `collection-sources.json`のID |
| `candidateType` | `news` / `tool` / `benchmark` / `evaluation` |
| `title` | 候補タイトル |
| `sourceUrl` | 元URL |
| `canonicalUrl` | 重複検知用の正規化URL |
| `sourcePublishedAt` | 原文公開日。不明な場合は`null` |
| `detectedAt` | 検知日 |
| `registeredAt` | 候補キュー登録日 |
| `suggestedStatus` | 原則`draft` |
| `duplicateCheck` | 重複検知結果 |
| `diffSummary` | 既存データとの差分概要 |
| `reviewStatus` | `pending` / `reviewing` / `accepted` / `rejected` |
| `reviewedAt` | レビュー日 |
| `reviewedBy` | レビュー者 |
| `promotedRecordType` | 採用後の正式データ種別。未採用時は`null` |
| `promotedRecordId` | 採用後の正式レコードID。未採用時は`null` |
| `promotedAt` | 正式データへ反映した日。未採用時は`null` |
| `notes` | 補足 |
| `changeLog` | 更新履歴 |

候補データ例:

```json
{
  "id": "candidate-news-openai-example-2026-001",
  "sourceId": "source-openai-news",
  "candidateType": "news",
  "title": "Example candidate for v1.1.2 collection design",
  "sourceUrl": "https://openai.com/news/",
  "canonicalUrl": "https://openai.com/news/",
  "sourcePublishedAt": null,
  "detectedAt": "2026-07-21",
  "registeredAt": "2026-07-21",
  "suggestedStatus": "draft",
  "duplicateCheck": {
    "status": "clear",
    "matchedIds": []
  },
  "diffSummary": "Design example. This is not a verified news item and must not be displayed as production news.",
  "reviewStatus": "pending",
  "reviewedAt": null,
  "reviewedBy": null,
  "promotedRecordType": null,
  "promotedRecordId": null,
  "promotedAt": null,
  "notes": "Do not promote without manual source review.",
  "changeLog": []
}
```

`update-candidates.json`には実候補のみを登録します。設計検証用の例は本書に残し、データファイルへは登録しません。

## 整合性ルール

| 条件 | ルール |
| --- | --- |
| `reviewStatus = accepted / rejected` | `reviewedAt`と`reviewedBy`を必須にする |
| `reviewStatus = pending` | `reviewedAt`と`reviewedBy`は`null`にする |
| `duplicateCheck.status = duplicate` | `matchedIds`を1件以上必須にする |
| `duplicateCheck.status = clear` | `matchedIds`は空配列にする |
| `promotedRecordType = null` | `promotedRecordId`と`promotedAt`も`null`にする |
| `suggestedStatus` | `draft`のみ許可し、`verified`は許可しない |

## 運用フロー

1. 公式ソースを確認する。
2. 新規または変更候補を`update-candidates.json`へ登録する。
3. URL、ID、重複、日付、差分概要を検証する。
4. 人が候補を確認し、重要度、CANAE影響メモ、カテゴリを判断する。
5. 採用する場合は`news.json`、`tools.json`、`benchmarks.json`などへ`draft`として反映する。
6. 別担当または責任者が公式内容を確認する。
7. 問題がなければ`verified`へ昇格する。
8. PRを作成し、CIとレビューを通してProductionへ反映する。

## 重複検知ルール

| 対象 | 判定キー |
| --- | --- |
| News | `sourceUrl`、正規化タイトル、公開日 |
| Tool | `toolId`、公式URL、会社名+製品名 |
| Benchmark | `toolId` + `benchmarkName` + `benchmarkVersion` |
| CANAE評価 | `toolId` + `evaluationVersion` + `useCase` |

重複が疑われる場合は自動採用せず、`duplicateCheck.status = "possible"`としてレビュー対象にします。

## 失敗時ルール

| 失敗 | 対応 |
| --- | --- |
| URL取得失敗 | 候補化しない。次回再確認 |
| 403 / bot block | 公式URLとして保持し、手動確認対象 |
| 日付不明 | `detectedAt`のみ登録し、公開日は未設定 |
| 重複疑い | `reviewStatus: pending`で停止 |
| 差分過多 | 人手レビュー必須 |
| 公式性不明 | `sourceType`を確定せず、採用しない |

## 通知方針

v1.1.2では通知の実装は行いません。

設計上は以下を候補にします。

- GitHub Issue
- GitHub PRコメント
- SlackまたはChatツール
- 手動レビューリスト

初期運用ではGitHub IssueまたはPRコメントを推奨します。

## Collection Helper Foundation

最初の実装PRでは、外部取得や候補登録には進まず、収集対象の選択ロジックだけを検証します。

| 項目 | 方針 |
| --- | --- |
| 入力 | `collection-sources.json`のみ |
| 対象選択 | `enabled: true`、`priority`、`cadence`で選択 |
| 実行方式 | dry-runのみ |
| 外部通信 | 行わない |
| 正本データ更新 | `news.json`、`tools.json`、`benchmarks.json`、`canae-evaluations.json`は変更しない |
| 候補登録 | `update-candidates.json`へは書き込まない |
| 通常出力 | `reports/collection-dry-run-report.local.json` |
| 固定サンプル | `reports/collection-dry-run-report.example.json` |
| コマンド | `npm run collection:dry-run` |

dry-runレポートには、選択された収集元、実行予定順、スキップ理由、外部通信なし、正本データ非更新、候補キュー非更新の状態を記録します。

通常実行ではAsia/Tokyo基準の当日日付を`generatedAt`へ入れます。再現試験では`--date=YYYY-MM-DD`を明示します。

`--priority`は`P0`、`P1`、`P2`のみ、`--cadence`は`daily`、`weekly`、`monthly`のみ許可します。未知の引数、不正な日付、空のリスト、`reports/`外への出力はエラーにします。

この段階で検証するのは収集対象の選択だけです。取得処理、候補生成、重複・差分検知、通知は別PRで扱います。

## External Fetch Foundation

次の実装PRでは、外部取得の挙動だけを2ソース限定で確認します。

| 項目 | 方針 |
| --- | --- |
| 入力 | `collection-sources.json`のみ |
| 初期対象 | `source-openai-news`、`source-github-changelog` |
| 実行方式 | read-only fetch dry-run |
| 外部通信 | 行う。ただし対象URLの読み取りのみ |
| 正本データ更新 | `news.json`、`tools.json`、`benchmarks.json`、`canae-evaluations.json`は変更しない |
| 候補登録 | `update-candidates.json`へは書き込まない |
| 通常出力 | `reports/external-fetch-report.local.json` |
| 固定サンプル | `reports/external-fetch-report.example.json` |
| コマンド | `npm run external-fetch:dry-run` |

fetchレポートには、HTTP status、最終URL、リダイレクト有無、要求元ホスト、最終ホスト、クロスドメインリダイレクト有無、HTTPS判定、リダイレクト安全判定、Content-Type、Content-Type許可判定、Content-Length、取得サイズ、所要時間、タイムアウト、404、429、エラー情報を記録します。

許可するContent-Typeは次の範囲に限定します。

- `text/html`
- `application/xml`
- `text/xml`
- `application/rss+xml`
- `application/atom+xml`

HTTP 403、404、429、500系などは取得結果としてレポートへ記録し、レポート生成自体に成功した場合はコマンドを正常終了します。通信不能、設定不正、入力不正、レポート保存失敗はコマンド失敗とします。レポートには`executionStatus`と`exitPolicy: report-generation-success`を記録し、HTTPエラーとコマンド実行失敗を分離します。

リダイレクト後の最終URLがHTTPSでない場合、または登録URLと異なるホストへ移動した場合は、取得結果を要手動確認として扱います。この段階では外部取得範囲の拡大を防ぐため、別ドメイン移動を自動的に安全とは判定しません。

OpenAI Newsはbot対策などにより403になる場合があります。この段階では失敗として扱い、手動確認または別取得方式の検討対象として記録します。

この段階で検証するのは外部取得の通信結果記録だけです。候補生成、`update-candidates.json`への書き込み、重複・差分検知、通知は別PRで扱います。

## Candidate Generation Foundation

次の実装PRでは、GitHub Changelogの200取得結果だけを対象に、候補生成の最小経路を確認します。

| 項目 | 方針 |
| --- | --- |
| 入力 | `reports/external-fetch-report.example.json`とGitHub Changelog HTML |
| 初期対象 | `source-github-changelog`のみ |
| 実行方式 | report-only candidate generation |
| 正本データ更新 | `news.json`、`tools.json`、`benchmarks.json`、`canae-evaluations.json`は変更しない |
| 候補登録 | `update-candidates.json`へは書き込まない |
| 通常出力 | `reports/candidate-generation-report.local.json` |
| 固定サンプル | `reports/candidate-generation-report.example.json` |
| コマンド | `npm run candidate:generate` |
| 検証 | `npm run validate:candidates` |

固定サンプルの`external-fetch-github-changelog.example.html`は、実取得HTMLの完全保存ではありません。抽出対象構造だけを再現した縮小fixtureです。GitHub側のDOM変更への耐性や実HTML全体での統合試験は、このFoundationでは完了条件に含めず、後続PRで扱います。

候補データには、`sourceId`、`candidateType`、`title`、`sourceUrl`、`canonicalUrl`、`sourcePublishedAt`、`detectedAt`、`summary`、`summarySource`、`summaryGenerated`、`reviewStatus`、`suggestedStatus`を含めます。要約は生成しないため、`summary`は`null`、`summarySource`は`none`、`summaryGenerated`は`false`に固定します。

この段階の候補は`update-candidates.json`投入前のレポートです。`reviewStatus`は`pending`、`suggestedStatus`は`draft`に固定し、`verified`昇格、自動公開、CANAE影響メモ生成は行いません。

重複・差分検知は未実行として`duplicateCheck.status: not-run`を記録します。同一HTML内の重複`canonicalUrl`は候補を1件に正規化し、スキップ件数を`skippedDuplicateCanonicalUrls`へ記録します。必須項目欠落、抽出失敗はレポートの`summary`と`extractionIssues`に記録します。将来の重複判定では候補IDではなく、`canonicalUrl`を正本識別子として優先します。候補IDはレポート内の内部識別子であり、同一記事性の主キーにはしません。

HTML抽出はFoundation段階では正規表現中心です。`<article>`単位で切り出せない場合、タイトルリンクclass変更、`datetime`欠落、相対URL、同一記事リンクの複数出現、ナビゲーションリンク混入が起き得ます。候補を無理に正式化せず、抽出不能・欠落・重複は`missingFields`、`extractionIssues`、`duplicateCanonicalUrls`へ記録します。

候補生成前には、External Fetch結果が`status: 200`、`ok: true`、`contentTypeAllowed: true`、`sourceId: source-github-changelog`、`finalHost: github.blog`、`finalUrl: https://github.blog/changelog/`であることを確認します。HTML入力は空でなく、GitHub Changelogリンクを含む必要があります。

OpenAI NewsはExternal Fetch Foundationで403として記録されているため、この段階では候補生成対象外とします。

## Duplicate & Diff Detection Foundation

次の実装PRでは、候補生成レポートと既存ニュースデータを照合し、重複・差分検知の最小経路を確認します。

| 項目 | 方針 |
| --- | --- |
| 入力 | `reports/candidate-generation-report.example.json` |
| 比較対象 | `news.json`または固定fixtureのニュースデータ |
| 実行方式 | report-only duplicate and diff detection |
| 正本データ更新 | `news.json`、`tools.json`、`benchmarks.json`、`canae-evaluations.json`は変更しない |
| 候補登録 | `update-candidates.json`へは書き込まない |
| 通常出力 | `reports/duplicate-diff-report.local.json` |
| 固定サンプル | `reports/duplicate-diff-report.example.json` |
| 比較fixture | `reports/duplicate-diff-news-fixture.example.json` |
| コマンド | `npm run duplicate-diff:detect` |
| 検証 | `npm run validate:duplicate-diff` |

照合の優先順位は、`canonicalUrl`完全一致、正規化URL一致、タイトル類似の順とします。正規化URLではプロトコルとホストを小文字化し、クエリ、フラグメント、末尾スラッシュを除外して比較します。タイトル類似は補助信号であり、自動採用や`verified`昇格には使いません。

各候補には`contentFingerprint`を付与します。方式はSHA-256、出力形式は`sha256:<hex>`です。入力文字列は、正規化済み`canonicalUrl`、正規化済みタイトル、`sourcePublishedAt`を改行区切りで連結した値とします。タイトル正規化では小文字化、記号除去、空白圧縮を行います。これは監査用のハッシュであり、同一記事性の主キーではありません。同一記事性の判定では、引き続き`canonicalUrl`を最優先します。

レポートには、候補ID、正規化URL、Content Fingerprint、照合状態、照合理由、既存レコードID、タイトル類似度、差分項目一覧を記録します。照合理由は後方互換用の単一`matchReason`に加え、複数条件を追跡できる`matchReasons`配列にも保持します。差分項目は`title`、`sourcePublishedAt`、`canonicalUrl`、`summary`の固定順で出力し、候補値と既存値を並べて確認できる形にします。

固定サンプルは、本番`news.json`を変更せずに検証するための最小fixtureです。完全一致、正規化URLのみ一致、タイトル差分、新規候補のケースを含めます。このfixtureはProductionニュースの正本ではなく、検証用データとしてのみ扱います。

この段階では、重複・差分検知結果を`update-candidates.json`へ自動反映しません。人手レビューUI、正式候補キューへの書き込み、`draft`登録、`verified`昇格、CANAE影響メモ生成は後続PRで扱います。

## Candidate Registration Foundation

次の実装PRでは、Duplicate & Diff Detectionレポートで新規候補と判定されたものだけを`update-candidates.json`へ登録します。

| 項目 | 方針 |
| --- | --- |
| 入力 | `reports/duplicate-diff-report.example.json`と`reports/candidate-generation-report.example.json` |
| 通常実行 | dry-run。`reports/`へレポートのみ出力 |
| 明示適用 | `--apply`指定時のみ`data/update-candidates.json`へ反映 |
| 登録先 | `data/update-candidates.json`または検証用`reports/`内candidate store |
| 登録対象 | `matchStatus: new`かつ元候補の`reviewStatus: pending` |
| 登録ステータス | `reviewStatus: pending`、`suggestedStatus: draft`に固定 |
| 登録日 | `registeredAt`を付与 |
| 重複拒否 | 同一候補ID、同一`canonicalUrl`、正規化`canonicalUrl`を拒否 |
| 正本データ更新 | `news.json`、`tools.json`、`benchmarks.json`、`canae-evaluations.json`は変更しない |
| 自動昇格 | `verified`昇格は行わない |
| 通常出力 | `reports/candidate-registration-report.local.json` |
| 固定サンプル | `reports/candidate-registration-report.example.json` |
| コマンド | `npm run candidate:register` |
| 検証 | `npm run validate:registration` |

登録時の`duplicateCheck.status`は`clear`に固定し、`matchedIds`は空配列にします。Duplicate & Diff Detectionで`duplicate`または`possible-duplicate`になった候補は登録せず、レポートに拒否・スキップ理由を記録します。

`data/`配下の候補ストアへ書き込む場合は`--apply`を必須にします。`--apply`なしで`data/`を指定した場合は失敗します。書込みは一時ファイルへ出力し、JSON再読込検証後にrenameする原子的更新とします。0件登録または全件拒否の場合は、`--apply`指定時でも候補ストアを書き直しません。

登録レポートには、入力件数、登録件数、スキップ件数、拒否件数、重複ID件数、重複`canonicalUrl`件数、正規化`canonicalUrl`重複件数、対象Candidate ID一覧、拒否理由、`storeChanged`、変更前後件数、入力・出力ハッシュを記録します。

この段階では、レビューUI、通知、`draft`から正式ニュースへの反映、`verified`昇格、CANAE影響メモ生成は行いません。自動化は候補キュー登録までに留め、公開判断は人手レビューに残します。
